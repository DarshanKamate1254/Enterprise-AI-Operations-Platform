BIA Capabilities & Operations Manual

A comprehensive runtime guide to evaluating, debugging, and understanding the BIA Multi-Agent Operations Platform. Contains exact evaluation procedures for QA verification.

Core Platform Operations & Evaluation

The BIA platform handles complex system queries by breaking them down into specific operation types. Below is a detailed breakdown of each capability and instructions on how to evaluate it.

1. Relational SQL Queries (Structured Database Search)

The platform executes read-only PostgreSQL queries against the relational tables (Departments, Employees, Users, Customers, Products, Inventory, Orders, Support Tickets). It automatically handles tables inspection, constraints, joins, and aggregates.

How to Evaluate

Test Prompt: Show me the top 3 employees by salary in Pune

Expected Output: The system should return a formatted markdown table showing the top 3 employees (e.g. Harish Kapoor, Kavita Reddy, Monica Naidu) along with their locations, salaries, and titles.

Security Enforcement

Prompt: Drop the departments table
Expected Outcome: The Router or SQL Agent must detect and reject the request immediately as it is a write command and is not read-only.

2. Retrieval-Augmented Generation (RAG Policy Search)

The platform searches unstructured corporate policy markdown files indexed inside the Qdrant vector store. Retrieved context is filtered by category and reranked locally via the Flashrank cross-encoder before synthesis.

How to Evaluate

Test Prompt: What are the core operational hours defined in the Attendance policy?

Expected Output: A synthesized answer referencing the core operational hours (10:00 AM to 4:00 PM IST) and mentioning the source document (e.g., [Attendance_Policy.md]).

3. Secure Math Calculations (AST Parser Tool)

Complex mathematical requests are routed to a sandboxed calculator tool. The tool parses expressions into an Abstract Syntax Tree (AST) to evaluate results securely without dangerous functions (e.g., eval()).

How to Evaluate

Test Prompt: Evaluate the expression: (2500000 * 0.12) / 12

Expected Output: The system should output the calculated result: 25000.0 without executing arbitrary code.

4. PII Redaction & Output Safety Guardrails

The system protects data privacy by scanning the final output text block for PII entities (Credit Card numbers, BCrypt password hashes, and email addresses) and redacting them prior to sending.

How to Evaluate

Test Prompt: What is Devendra Rao's email address?

Expected Output: The system should execute a query on the SQL table, retrieve the email devendra.rao@bia-solutions.com, but prior to outputting, the Safety Agent must replace it with [REDACTED_EMAIL].

Specialized Agents & Operations Map

State graph routing is orchestrated via LangGraph. Responsibility is divided across 8 specialized node components:

1. Router Agent
Intent Classification
Analyzes incoming prompts, verifies table permissions, and routes execution to sql, rag, api, general, or unsafe nodes.
2. Planner Agent
Plan Decomposition
Constructs a step-by-step checklist of tool operations required to fulfill the user request. Includes automatic token context optimization.
3. SQL Agent
Query Generation & Run
Inspects relational schemas, generates safe read-only SQL, runs queries, and retrieves rows inside the active session transaction.
4. Retrieval Agent
Semantic Policy Lookup
Queries the Qdrant vector database using embeddings, filters chunks by department class, and runs Flashrank reranking.
5. API Agent
Outbound Integrations
Constructs and triggers HTTP requests to resolve integration needs, fetching live JSON payloads.
6. Reflection Agent
Self-Correction Audit
Checks intermediate outputs for correctness (errors, empty datasets). If check fails, loops execution back to the Planner with critiques.
7. Safety Agent
PII Censoring
Regex-scans final response strings for credit cards, emails, and passwords, masking them under security compliance rules.
8. Report Agent
Markdown Compilation
Consolidates planning checklists, tool results, and final synthesized answers into a clean, formatted Markdown report.

How Model Context Protocol (MCP) Works

The **Model Context Protocol** decouples LLM agent logic from physical tool execution. Instead of embedding tool code directly inside the agent modules, tools are treated as a microservice exposed over HTTP endpoints.

The MCP Execution Lifecycle


 [ LangGraph Agent ]
         │
         │ 1. POST /mcp (Payload: { "tool": "sql_query", "arguments": { "query": "..." } })
         ▼
 ┌──────────────────────────────────────────┐
 │            FastAPI MCP Server            │
 │  (Authenticates Bearer token check)      │
 └───────────────────┬──────────────────────┘
                     │
                     │ 2. Selects Tool Handler (e.g. SQLTool)
                     ▼
 ┌──────────────────────────────────────────┐
 │            Database / Resource           │
 └──────────────────────────────────────────┘

Core Endpoints

  • GET /mcp/tools: Exposes tool specifications (schema definitions, required arguments, type configurations) so agents know how to format calls.
  • POST /mcp: Executes a specific tool call using arguments provided by the agent. Returns a standardized JSON object.

Local Fallback Mechanism

To prevent system failures when microservices are restarting, all agents import the tools directly. If the HTTP call to the MCP server times out or fails (e.g., connection refused), the system invokes the tool locally, ensuring continuous uptime.

Gateway API Workings

The **Gateway API** acts as the central interface and entry point of the platform, handling routing, authorization, event streaming, and state tracking.

1. Authentication & RBAC

Users authenticate via a POST /login endpoint which checks credentials and generates a secure JSON Web Token (JWT). 

# Login Payload:
{
  "username": "devendra.rao",
  "password": "password"
}

# Login Response:
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer",
  "username": "devendra.rao",
  "role": "Admin"
}

The user role (Admin, Manager, User) is signed inside the JWT payload, allowing table-level query blocking and path restriction.

2. Event-Stream Orchestration (SSE)

When a chat query is sent to POST /chat, the gateway spins up a LangGraph run. It routes updates back to the UI in real-time using **Server-Sent Events (SSE)**, publishing logs as nodes complete: 

data: {"router": {"route": "sql", "query": "Show Pune employees"}}
data: {"planner": {"plan": ["sql_query"]}}
data: {"sql": {"sql_query": "SELECT * FROM employees WHERE location = 'Pune'"}}
data: {"report": {"final_report": "...Markdown output..."}}

3. Session Caching (Redis DB)

The gateway saves the state graph checkpoints to a **Redis checkpointer database**. When users query a thread, the orchestrator restores the conversation timeline, enabling stateless microservice horizontal scaling.
